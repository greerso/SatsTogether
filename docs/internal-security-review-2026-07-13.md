# Internal Security Review — SatsTogether prototype (2026-07-13)

## Scope / non-claims

This is a **Claude-run internal review**, a **zero-budget substitute** for an
external audit. It is **NOT** a paid audit, **NOT** an external audit, and
**NOT** an auditor sign-off. No third party reviewed this code. It does **not**
satisfy the Phase 3 exit gate (see `docs/audit-checklist.md`,
`docs/production-roadmap.md`), which still requires a real external audit and a
funded bounty.

The target is a **testnet-only prototype**: ephemeral in-memory sessions, no
real funds, no custody, mock/design-stage protocol (BitVM2, covenants, yield
proofs are not implemented). Severities are rated against **that** system — a
public demo server with no user money at risk — not against a mainnet
custody product.

**Reviewed at:** tag `audit-v0.1.0-prep` + current `main`
(`feat/p3-claude-internal-review`).

**Primary targets reviewed (full read):**
`web/server.ts`, `web/session-store.ts`, `web/public/app.js`, `sim/ledger.ts`,

## Method

Adversarial four-lens read (Correctness, Internal Consistency, Completeness,
Implementation Risk). Candidate findings were generated by a separate reviewer
model (the `advisor` tool was unavailable this run; the review was delegated to
a `fable` reviewer subagent with a self-contained, conclusion-free prompt), then
each candidate was verified against the code before inclusion. High-confidence,
behavior-preserving fixes for the memory-exhaustion vectors were applied and
smoke-tested. Focus areas per the engagement: XSS, DoS, authz, ledger
accounting, draw fairness, mainnet footguns, info leaks, race conditions.

Verification of applied fixes: `npm test` → 77 pass / 0 fail / 1 skip; live boot
smoke — oversized (2 MB) POST body → `400`, valid import → `200`, `/health` →
`200`.

### Detail on fixed items

**ST-1 (fixed).** `clientIp` uses the attacker-controlled first `X-Forwarded-For`
value as the throttle key, and `rateHits` was never pruned — a request loop with
rotating forged headers minted a permanent map entry each time → OOM. Fix bounds
the map (`RATE_MAP_MAX=10_000`) and prunes entries past the longest cooldown
(deleting an already-expired throttle is semantically safe). This removes the
*unbounded-growth* memory vector regardless of XFF spoofing. The spoofing itself
(throttle bypass, ST-3) is a separate, unfixed concern.

**ST-2 (fixed).** `readBody` buffered the entire request body with no ceiling
before `JSON.parse`; `/api/session/import` is unauthenticated and was
un-rate-limited, so a multi-GB POST was a trivial OOM. Fix rejects bodies over
1 MB mid-stream (snapshots are far smaller). Verified: 2 MB body → `400`.

### Detail on key open items

**ST-3 / ST-4** are the two that matter most for the *eventual real protocol*:

- **ST-3** is rooted in trusting `X-Forwarded-For`. A correct fix depends on the
  deploy topology (Traefik/Cloudflare in front): naively switching to
  `socket.remoteAddress` collapses all users behind the proxy to one IP and makes
  throttling global. Recommend a trusted-proxy-hop configuration (or Cloudflare
  `CF-Connecting-IP`) rather than a blind code change. Left open deliberately.

- **ST-4** is the flagship property for a prize protocol: **draw fairness**. The
  offline mix (`placeholderMix`) is non-cryptographic (documented), and because
  the tip block hashes are already public when the user commits their own
  `userSeed`, the commit-reveal adds no entropy the caller lacks — the caller can
  grind `userSeed` offline and pick winners. In the current prototype this only
  lets a user win *their own* single-session sim yield (no cross-user funds), but
  the `/api/status` honesty line "Session draws require seed commit-reveal"
  (`web/server.ts:177`) **overstates** the protection and should be softened.
  Real fairness needs consensus-side entropy the participant cannot pre-compute
  (future block hash / VRF / BitVM2), per `docs/bitvm-challenge-game.md`.

## Explicit non-issues (checked, verified safe)

- **XSS (`app.js`)** — every user-controlled string reaching an `innerHTML` sink
  is escaped via `esc()` (`app.js:122,144,150,216`); all other interpolated
  values are numeric `bigint`/`number` strings (round-tripped through
  `BigInt()`/`Number()`), so they can't carry markup. Avatars/titles/toast use
  `textContent`/property assignment.
- **CSRF** — state-changing routes are POST behind a `SameSite=Lax; HttpOnly`
  cookie (`server.ts:97`); Lax blocks cross-site POST cookie attachment.
- **Cross-session authz** — sessions keyed by unguessable `randomUUID()` bearer
  cookie; `getOrCreateSession` never adopts a client-supplied unknown id. No
  enumeration path; `account` strings are intra-session labels, not identities.
- **`scripts/deploy-mainnet.sh`** — refuse-by-default: prints "NOT implemented"
  and `exit 1` even on `y`. No mainnet footgun.
- **Governance crypto** — `Bip322Signer` fails closed (`verifyMessage` → `false`,
  `signMessage` throws); `MockSigner` and `voting.ts` (non-sybil-resistant) are
  honestly labeled. No false security claims in code.
- **`Dockerfile`** — runs as `USER node`, no secrets baked, testnet-only default.

## Residual risks

- **Availability, not confidentiality/integrity.** After ST-1/ST-2, the
  remaining DoS surface (ST-3 explorer amplification, ST-5 session flood, ST-7
  mint cost) can still degrade or crash the free demo server. Acceptable for a
  no-funds prototype; **not** acceptable for anything holding value.
- **Draw is fully manipulable** by design (ST-4). The protocol's core
  no-loss/fair-draw guarantees are **unproven and not implemented**. Nothing here
  should be read as evidence the real draw is fair.
- **In-memory, single-instance, ephemeral.** No durability, no multi-instance
  coordination; a restart wipes all state. This is intended for the prototype.
- **This review is not independent assurance.** One model read the code; it can
  miss classes of bugs a funded, tooled audit would catch.

## Recommended next steps

1. **P3 exit still unmet.** A **paid external audit** and a **funded bug bounty**
   remain mandatory before any mainnet / real-funds path. This internal review
   does not substitute for either.
2. Soften the `/api/status` commit-reveal honesty line (ST-4) so it does not
   imply the draw is unpredictable. **Done 2026-07-13** (status + UI copy).
3. Rate-limit import endpoint. **Done 2026-07-13**.
4. Decide trusted-proxy handling (ST-3) as part of deploy hardening, not a blind
   code change.
5. If the prototype must survive hostile traffic: rate-limit session creation
   and remaining mutate endpoints, switch eviction to LRU (ST-5), and add
   snapshot-consistency validation on import (ST-6).
6. Carry open ST items into funded review when budget allows.
